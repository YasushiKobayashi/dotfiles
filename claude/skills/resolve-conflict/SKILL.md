# Git コンフリクト解消

## 概要

マージ/リベースで発生したコンフリクトを、両ブランチの意図を理解した上で安全に解消する。

**指定されたファイル**: $ARGUMENTS（未指定の場合は全コンフリクトファイルを対象）

---

## 現在の状態

- ブランチ: !`git branch --show-current`
- マージ/リベース状態: !`git status | head -3`
- コンフリクトファイル一覧: !`git diff --name-only --diff-filter=U 2>/dev/null || echo "コンフリクトなし"`

---

## タスク

### 1. コンフリクトファイルの特定

```bash
git diff --name-only --diff-filter=U
```

引数でファイルが指定されていればそのファイルのみ対象。未指定なら全コンフリクトファイルを対象とする。

### 2. 各ファイルについて: 両ブランチの変更意図を分析

**以下の情報を収集し、各ブランチの「本質的な変更意図」を把握する。**

#### a. コンフリクト箇所の確認

対象ファイルを Read で読み、コンフリクトマーカー (`<<<<<<<`, `=======`, `>>>>>>>`) の箇所を特定する。

#### b. ベースブランチの特定

明示的な指示がなければ `main` をベースブランチとして使用する。ユーザーから特定ブランチへのマージと指示があればそのブランチを使用する。

#### c. HEAD側の変更意図

```bash
# HEADブランチでの該当ファイルのコミット履歴
git log --oneline <base_branch>..HEAD -- <file>
# 各コミットの変更内容
git show <commit> -- <file>
```

#### d. 相手ブランチの変更意図

```bash
# 相手ブランチでの該当ファイルのコミット履歴
git log --oneline HEAD..<base_branch> -- <file>
# 各コミットの変更内容
git show <commit> -- <file>
```

### 3. 解消方針の策定

両ブランチの変更を分析し、以下の方針で解消案を作成する:

- **両立可能**: 両方の変更を取り込む（最優先）
- **一方が包含**: 片方の変更がもう片方を含んでいる場合、包含する側を採用
- **競合**: 同じ箇所を異なる意図で変更している場合、ユーザーに判断を仰ぐ

### 4. ユーザーへの確認

**必ず AskUserQuestion で確認を取ってから実行する。**

確認時に提示する情報:

- コンフリクト箇所ごとに、HEAD側の意図 / 相手ブランチ側の意図 / 解消案を表形式で示す
- 推奨案がある場合は `(Recommended)` を付与
- 解消後のコードをプレビューで提示

### 5. コンフリクト解消の実行

ユーザーの承認後、Edit ツールでコンフリクトマーカーを含む箇所を解消後のコードに置換する。

```
# Edit で以下を置換（コンフリクトマーカーを含むブロック全体をファイルから正確にコピーして指定すること）
old_string: （<<<<<<< HEAD から >>>>>>> <branch> までの全文を一字一句そのまま指定）
new_string: <解消後のコード>
```

### 6. 解消の検証

```bash
# コンフリクトマーカーが残っていないか確認
grep -n '<<<<<<<\|=======\|>>>>>>>' <file>

# ステージング
git add <file>
```

### 7. 全ファイル完了後の確認

```bash
# 残りのコンフリクトファイルがないか確認
git diff --name-only --diff-filter=U
```

全ファイルの解消が完了したら、ユーザーにその旨を報告する。

---

## 注意事項

- コンフリクト解消は**必ずユーザー確認を経てから実行**する
- `git checkout --ours` / `--theirs` による一括採用は原則禁止（変更意図の分析を省略してしまうため）
- 解消後にコミットは行わない（ユーザーが `/commit` で行う）
