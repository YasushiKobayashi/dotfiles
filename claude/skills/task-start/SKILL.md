# Task Auto (task.\*.md 駆動の単発タスク遂行)

## 概要

ユーザーが指定した `task.*.md`、または **実行ディレクトリ (cwd) 直下に新規作成する** `task.*.md` を **そのタスクの SSOT（メモ・進捗・成果物リンク集）** として扱い、1 タスクを完遂するための skill。

**ファイルの場所**:

-   **タスクファイルは必ずしもリポジトリ直下ではない**。`front/`, `go/`, `doc/` 等のサブディレクトリ配下に置かれることもある。
-   **既存ファイル**: ユーザーが引数 / 会話文脈で示したパスをそのまま使う（絶対パスでも相対パスでも可）。場所を勝手に動かさない。
-   **新規作成**: `pwd` で確認した **実行ディレクトリの直下** に作る（例: cwd が `front/` なら `front/task.auto.md`）。ユーザーが特に指定しない限り、上位ディレクトリ・別サブツリーへ作らない。

**命名規約**:

-   ユーザーのグローバル `.gitignore` で `task.*.md` がすべて無視される運用。**Claude が自動で作成するタスクファイルは必ず `task.*.md` パターンに従う**。
-   既定ファイルは `task.auto.md`。issue 駆動なら `task.<issue番号>.md`、トピック駆動なら `task.<kebab-slug>.md` を使う。
-   それ以外のファイル名（例: `TASK.md`, `notes.md`, `todo-1884.md`）は **使わない**。gitignore から漏れて誤って commit される事故を防ぐため。

**コアな考え方**:

-   `task.auto.md` は「**作業ログ**」と「**チェックリスト**」と「**最終的なナレッジ**」を 1 ファイルに集約するための器。
-   セッションが落ちても、別エージェントに引き継いでも、このファイルを読めば「何をしようとしていて、どこまで進み、なぜそうしたか」が分かる状態を維持する。
-   完了後のクリーンアップ（削除 / PR description への転記）は **ユーザーが手で行う**。skill 側では消さない。

**引数**: $ARGUMENTS — 次のいずれかを受け取る前提:

| 引数のタイプ            | 判定方法                                                              | 対応モード                                                                                                                                              |
| ----------------------- | --------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **タスクファイルパス**  | `task.*.md` パターンに合致 / 既存ファイルとして読める                 | **続行モード**（§1）                                                                                                                                    |
| **GitHub issue/PR URL** | `github.com/<owner>/<repo>/(issues\|pull)/<num>` または `#<num>` 形式 | **新規作成モード**。`gh issue view` / `gh pr view` でタイトル・本文・ラベルを取得して雛形に流し込む                                                     |
| **Notion ページ URL**   | `notion.so/...` / `*.notion.site/...`                                 | **新規作成モード**。Notion MCP (`mcp__Notion__*`) でページ本文を取得して雛形に流し込む。MCP が認証されていない場合は `mcp__Notion__authenticate` を案内 |
| **自由記述 / なし**     | 上記いずれにも当てはまらない                                          | **新規作成モード**。AskUserQuestion でタイトル・チェックリスト粒度をユーザーに確認しつつ作る                                                            |

判定は引数文字列の先頭を見るだけで十分。複数渡された場合（例: タスクファイルパス + issue URL）は **ファイルを SSOT として扱い、URL は冒頭の参照リンクに追記** する。

## ワークフロー

### 0. モード判定

冒頭で次のいずれかに振り分ける。**判定にあたっては引数のパス指定 > cwd 直下の存在 > リポジトリ内の他の場所** の優先順位で対象を決める。

| 状況                                                  | モード               | 最初にやること                                                                      |
| ----------------------------------------------------- | -------------------- | ----------------------------------------------------------------------------------- |
| 引数で具体的なパス（`path/to/task.foo.md`）が渡された | **続行（指定優先）** | そのパスを Read。存在しなければユーザーに確認                                       |
| 引数が GitHub issue/PR URL                            | **新規作成**         | `gh issue view` / `gh pr view` で内容取得 → 雛形へ流し込み → AskUserQuestion        |
| 引数が Notion URL                                     | **新規作成**         | Notion MCP でページ取得 → 雛形へ流し込み → AskUserQuestion                          |
| 引数なし、cwd 直下に `task.*.md` が 1 つ              | **続行**             | ファイル全文を Read。未完了 `[ ]` を抽出                                            |
| 引数なし、cwd 直下に `task.*.md` が複数               | **要確認**           | 一覧を提示し AskUserQuestion で選んでもらう                                         |
| 引数なし、cwd 直下には無いがサブツリー / 上位に存在   | **要確認**           | 検出されたパスを提示し「これを使うか / cwd 直下に新規作成するか」を AskUserQuestion |
| 引数が自由記述だけ                                    | **新規作成**         | 概要をタイトルに使う。詳細は AskUserQuestion で補う                                 |
| 何もない                                              | **要確認**           | AskUserQuestion で issue URL / Notion URL / タスク概要を聞き、新規作成モードへ      |

**禁止事項**: §注意事項の「ファイル操作の禁止事項」を必ず守る（上書き禁止 / 命名規約 / cwd 直下のみ / commit しない / 冒頭 2 行を消さない）。

### 1. 続行モード: 既存ファイルの読解

対象の `task.*.md` を Read で全文読み、以下を抽出する:

-   **タイトル**: 最初の `# ...` 行（多くの場合 issue タイトル）
-   **参照 URL**: issue / PR / Figma / docs リンク
-   **チェックリスト**: `- [ ]` / `- [x]` の項目
-   **`## メモ` セクション**: 既存の原因分析・修正方針・検証メモ
-   **冒頭の指示文**: 「メモはこのファイルに」「重要ドキュメントは適宜残して」等のメタ指示

未完了 `[ ]` 項目を `TodoWrite` に 1 タスク 1 項目で投入する（タイトルはチェックリスト原文）。既に `[x]` の項目は登録しない。

**現在地の把握 (再開時に必須)**: チェックリストだけ見て即実装に入らない。`## メモ` を読んで次を整理してから着手する:

-   最後に書かれた仮説 / 試行は何か（`### 原因` / `### 修正` の末尾）
-   既に却下した案 / 試して外したアプローチはあるか（`### 意思決定の理由`）
-   「### 残課題」「### TODO」「### 行き詰まり」等の動的な見出しに残された未消化メモ
-   直近の `[x]` チェック項目と `### 修正` の整合（書き忘れ・齟齬があれば追記）

これらを 1-2 行でユーザーに「現在地サマリ」として返してから次サブタスクへ進むと、認識ズレが起きにくい。

### 2. 新規作成モード: 雛形の作成

#### 2-a. 配置場所とファイル名の決定

**配置場所**: 原則として **実行ディレクトリ (cwd) の直下**。`pwd` で確認した場所にそのまま作る。

-   例: cwd が `~/go/src/.../iesapuri/front` なら `front/task.auto.md`
-   リポジトリルートや別サブツリーへの配置は、ユーザーが明示的に指示した時だけ。
-   既存ファイルが別の場所にあって続行モードになる場合は、§1 の対象パスをそのまま使い、cwd 直下に複製しない。

**ファイル名**: **必ず `task.*.md` パターン** を使う。優先順位:

1. issue 駆動: `task.<issue番号>.md`（例 `task.1884.md`）
2. トピック駆動 / 一時的な作業: `task.<kebab-slug>.md`（例 `task.quotation-realtime.md`）
3. 何も決められない / アドホック: `task.auto.md`

判断が割れる場合は AskUserQuestion で候補を 2-3 個提示してユーザーに選んでもらう。既に同名ファイルがある場合は別名を提示する（上書き禁止）。

> `task.*.md` はユーザーのグローバル `.gitignore` で除外される運用のため、PR に混入する心配はない。逆に **このパターンから外れた名前を作ると commit に紛れ込むリスク** があるため厳守。

#### 2-b. 引数ソースからの情報取り込み

雛形を書き始める前に、引数の種類に応じて **タイトル / 本文 / ラベル / 関連リンク** を取得しておく。

| 引数種別                          | 取得コマンド / ツール                                                 | 取り込む項目                                               |
| --------------------------------- | --------------------------------------------------------------------- | ---------------------------------------------------------- |
| GitHub issue                      | `gh issue view <num> --json title,body,labels,assignees,url,comments` | タイトル、本文要約、ラベル、最新コメントのうち仕様確定情報 |
| GitHub PR                         | `gh pr view <num> --json title,body,url,headRefName,baseRefName`      | タイトル、PR description、対象ブランチ                     |
| `owner/repo#N` 形式や別リポジトリ | `gh issue view <num> --repo <owner>/<repo> ...`                       | 同上                                                       |
| Notion ページ                     | `mcp__Notion__*`（ページ取得系ツール）                                | ページタイトル、本文の見出し構造、TODO / リスト項目        |
| 自由記述                          | （取得無し）                                                          | ユーザー発話そのまま                                       |

**重要**:

-   Issue/PR の本文をそのままコピペせず、**チェックリスト化できる単位に要約** する。長文は `### 仕様` / `### 参考` 見出しに別途貼る。
-   Notion ページの `to_do` ブロックは `- [ ]` / `- [x]` にマッピングして取り込める（チェック済み状態も保持）。
-   ネットワーク取得が失敗した場合は AskUserQuestion でフォールバック（手入力 / 後で再試行）。

#### 2-c. 雛形

取得済み情報を元に、次のテンプレートで作る:

```markdown
<ファイル名> を見てタスク進めて
タスクの実行におけるメモはこのファイルに行って、実装に進めるにあたって重要なドキュメントは適宜残してください。

# <タイトル>

<引数で受け取った URL（GitHub issue/PR、Notion ページ、Figma、関連 docs など）を箇条書き>

-   [ ] <最初に思いつくサブタスク>
-   [ ] <次のサブタスク>
-   [ ] <検証 / e2e / spec 更新>

---

## メモ

### 原因

### 修正

### 検証

### 意思決定の理由
```

雛形の **チェックリストはあくまで仮**。実コードを軽く偵察（Read / Grep）してから「この粒度で進めて良いか」を AskUserQuestion で確認する。

-   不明な部分は `<TBD>` のままにしておく
-   仮のチェックリスト項目に確信が無いことを明記してから尋ねる

ユーザー承認後に Write で保存し、続行モードと同じ手順で `TodoWrite` に投入する。

### 3. サブタスクの逐次実行

`TodoWrite` の項目を 1 つずつ `in_progress` にして実行する。**1 サブタスクが完了したら即座に**:

1. **実装系サブタスクなら**プロジェクト規約の lint / typecheck を回す（例: `mise run go:lint` / `mise run front:lint` / `mise run pre-commit` 等。プロジェクトの `CLAUDE.md` / `AGENTS.md` / `README` で指定されたコマンドを使う）
2. `TodoWrite` を `completed` に
3. 対象 `task.*.md` のチェックリスト該当行を `[ ]` → `[x]` に Edit
4. 必要なら `## メモ` に下記 4 項目のうち判明した部分を追記（後述 §4）

lint が失敗したらサブタスクは **未完了のまま**。`### 残課題` に「lint で X が出ている」を 1 行残してから次に進むか修正する。

**重要**: チェックリストの更新を「全部終わってから一気に」ではなく **1 項目ごとに即時** に行う。理由はセッション中断時の復旧性。

**Edit でチェック行を `[x]` 化する際の注意**:

-   `- [ ]` だけだと `old_string` が unique にならず Edit が失敗する。必ず **アイテム名込み** で指定する（例: `- [ ] e2eテスト更新` → `- [x] e2eテスト更新`）
-   それでも複数該当する場合は前後 1-2 行をコンテキストに含めて unique にする
-   `replace_all` は使わない（他のチェックボックスも巻き込むため）

新たに判明したサブタスクがあれば:

-   既存チェックリストに `- [ ]` で追記（位置は実行順序に沿った場所）
-   同時に `TodoWrite` にも追加
-   1 項目だけの軽い branch out なら、サブ箇条書き `  - [ ]` でネストしても良い

### 4. メモセクションの育て方

`## メモ` 配下に **最低限**次の 4 見出しを置き、判明し次第埋めていく:

| 見出し               | 書く内容                                                                                 | 例                                                                                                                               |
| -------------------- | ---------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `### 原因`           | なぜその不具合 / 課題が発生していたかの **root cause**。表面的な症状ではなく構造的な理由 | 「`form.watch()` がネスト配列の参照を更新しないため `useMemo` がキャッシュされる」                                               |
| `### 修正`           | どのファイルをどう変えたか。**ファイル名と変更要点** を必ず書く                          | 「`QuotationSingle.tsx`: `form.watch()` → `useWatch({ control, name: 'details' })`」                                             |
| `### 検証`           | テスト・手動確認・スナップショット更新など、**どう壊れていないか** を再確認した手段      | 「`QuotationSingle.spec.tsx` 新規。fix を stash で外すと spec 失敗を確認」                                                       |
| `### 意思決定の理由` | 採用案の根拠、却下した代替案、スコープ判定                                               | 「multiple モード側と同じ修正パターンに揃えるため `useWatch` を採用。`useEffect + setState` 案はレンダリング数が増えるので却下」 |

**書き方の原則**:

-   **未来の自分 / 別エージェントへの引き継ぎ書** だと思って書く。「これを読めば PR description が書ける」を目標に。
-   ファイル参照は `path/to/file.ts:42-58` 形式で行番号付き。
-   引用するなら ` ```ts ``` ` のコードフェンス。
-   既出の内容を消さない（誤りなら取り消し線 `~~...~~` か「修正前: ...」「修正後: ...」で並置）。
-   TODO が残ったら「### 残課題 / TODO」見出しを追加して `- [ ]` で書き残し、チェックリスト本体にも転記する。

### 5. 「実装に進めるにあたって重要なドキュメント」の残し方

冒頭指示文 (`実装に進めるにあたって重要なドキュメントは適宜残してください`) を **コードを書く前 / 書きながら** 実践する:

-   関連する既存実装の参考箇所 → `### 参考` 見出しに `path/to/file.ts:42` + 1 行コメント
-   仕様の根拠ドキュメント → `### 仕様` 見出しに `doc/...md` リンク + 関連節
-   動作確認のための URL / Story 名 / E2E スペック名
-   ロールバック手順や手動確認手順（環境変数を一時的にいじる場合など）

書きすぎは害ではない。**実コードに残しても邪魔な「なぜ」は task.\*.md 側に逃がす** くらいの感覚で OK（どうせ gitignore で commit されない）。

### 6. 着地確認 (タスク完了直前)

すべての `- [ ]` が `- [x]` になったら、ユーザーに渡す前に以下を確認:

-   [ ] チェックリストに未消化項目が残っていないか（`grep -n '\[ \]' <ファイル名>`）
-   [ ] `## メモ` の 4 見出し（原因 / 修正 / 検証 / 意思決定の理由）が **すべて埋まっている** か。判らなかった項目は「不明」「該当なし」と明記
-   [ ] 関連 issue / PR の URL が冒頭に記載されているか
-   [ ] lint / typecheck / 該当範囲のテストを実行済みか（プロジェクトの規約に従う。`mise run pre-commit` 等）
-   [ ] 修正したファイルが `task.*.md` のメモから辿れる（ファイル名漏れがない）か

これらは **完了報告に先立つチェックリスト**。1 つでも欠ければ完了と言わない。

### 7. ユーザー向け最終報告

**そのまま PR description に貼れる体裁** でまとめる。`## Summary` / `## Test plan` のような PR 規約に近い構造にしておくと、ユーザーが後段で `gh pr create` するときに転記コストが最小化される。

```markdown
## Summary

<タイトル / 1-3 行で「何をなぜ変えたか」>

関連: <issue / Notion / Figma URL>

## 原因

<root cause を 1-3 行。task.\*.md の `### 原因` から要約>

## 変更内容

-   path/to/file.ts:42-58 — <要点>
-   path/to/other.ts:10-20 — <要点>

## Test plan

-   [x] <追加・更新したテストファイル名 + ケース>
-   [x] <手動確認した URL / 手順>
-   [x] lint / typecheck 通過 (`mise run pre-commit` 等)

## 残課題

-   なし
-   or: <残課題>: 別 issue #N で追跡 / 後続 PR で対応 等

---

タスクファイル `<相対パス>` を最新化済み（gitignore 対象なので commit には含まれません）。

次のアクション:

1. 差分を確認 (`git diff`)
2. `/commit` で commit 作成
3. push / `gh pr create` で PR 作成（必要なら上記 Summary をそのまま description に貼り付け）
```

タスクファイルの削除・コミット・PR description への転記・push は **行わない**。ユーザーの操作を待つ。

## 注意事項

**ファイル操作の禁止事項** (§0 の禁止事項を含む単一の出典):

-   既存 `task.*.md` を **無断で上書きしない**。中身があるなら必ず AskUserQuestion を経る
-   新規作成時に **`task.*.md` 以外の名前を使わない**（gitignore 漏れ事故防止）
-   **新規作成は cwd 直下のみ**。`pwd` で確認してから Write する。他の場所に作りたい場合は AskUserQuestion を挟む
-   skill 側で `task.*.md` を **コミット / ステージング / 削除しない**。`git add` も含めユーザー操作に委ねる
-   **冒頭 2 行（指示文）を消さない / 書き換えない**。ユーザーの運用ルール

**書き方の原則**:

-   **チェックリストの即時更新を守る**。バッチ更新は中断耐性を下げる。1 項目完了 → ファイル Edit → 次
-   **メモは積み上げ式**。古い分析が誤りだったとしても「修正前の理解」として残し、新しい結論を併記する。後で「なぜ最初その方向で進んだか」が辿れる方が価値が高い
-   **TodoWrite とタスクファイルは併走**。TodoWrite はセッション内の作業順序管理、`task.*.md` は永続ナレッジ。両方に同じ項目が出るのが正常
-   **新規作成モードで雛形を保存する前に必ず AskUserQuestion** を経る。チェックリストの粒度はユーザーの想定に合わせる
-   **タスクが大きすぎる場合は分割案を提示**。`task.<topic>.md` を `task.<topic>.<sub>.md` 等に分けるかを AskUserQuestion で確認。skill 側で勝手に分割しない
-   **`.gitignore` 漏れ確認**（`git check-ignore <ファイル名>`）。グローバル無視に含まれていない異常系では「コミット前に整理した方が良いか」をユーザーに報告

## よくある失敗

-   **チェックリストを全部済ませてから一気にメモを書く**: 途中の試行錯誤や却下案が抜け落ちる。1 項目ごとに少しずつ書く方が結果的に楽。
-   **「原因」を症状で書いてしまう**: 「合計が表示されない」は症状。「`form.watch()` がネスト配列の参照を更新しない」が原因。両者を区別する。
-   **修正セクションにファイル名を書かない**: 後で PR description を書くときに grep し直す羽目になる。実装した瞬間に書く。
-   **「検証」が空のまま完了報告**: テストを書いていない / 手動確認していないのと同義。何らかの確認手段を必ず書く（無理ならその旨を「未検証」と明記）。
-   **既存メモを上書きして履歴を消す**: 誤った仮説でも残す。新しい結論は併記。
-   **タスクファイルを勝手に削除 / コミット**: ユーザーの運用領域を侵害。完了後はファイルを残してユーザーに渡す。
-   **新規作成モードで確認なしに雛形を書き殴る**: チェックリストの粒度がユーザーの想定とズレる。1 度確認を入れる。
-   **issue URL を冒頭に書き忘れる**: 後でコンテキストを辿れなくなる。最初に必ず貼る。
-   **cwd ではなくリポジトリルートに勝手に作る**: モノレポでサブツリー単位で作業しているとき、ルートに作ると別作業のメモと混ざる。`pwd` を確認してから Write する。
-   **既存ファイルを cwd 直下にコピーしてしまう**: 続行モードでは元の場所をそのまま使う。複製すると同期が崩れる。
