# task-start: タスクファイルの具体例

`SKILL.md` の雛形（§2-c）は最小の骨組みだけなので、ここでは **実際に埋まった状態のゴールイメージ** を示す。
パス・issue 番号・URL はすべてプレースホルダ（`<owner>/<repo>` 等）なので、自分のプロジェクトに読み替えて使う。

共通の慣習（全例で守る）:

-   **冒頭 2 行の指示文は必ず付ける**（運用ルール。消さない・書き換えない）。
-   **タイトルは H1**。issue 駆動なら末尾に `(#1234)` を付けるか、直下に Issue リンクを置く。
-   **タイトル直下に `## 関連リンク` セクションを置く**（ページ最上部）。Issue / PR / 関連 issue・サブタスク / Notion / Figma / 設計 SSOT doc などを集約する。**タスク中に PR や issue を新規作成したら、その URL をここに即追記**（後回しにしない）。
-   **チェックリストは `## チェックリスト` 見出しの下**。フラット、または `### Phase X` でグループ化。**箇条書きのネストは避ける**（グループ分けは `###` 見出しで行う）。
-   **`## メモ` の見出しはタスク種別に合わせて増減してよい**。`### 原因 / 修正 / 検証 / 意思決定の理由` を最小セットとし、必要に応じて `### 仕様` `### Figma 比較メモ` `### 設計判断ログ` `### 残課題` `### 参考` 等を足す。
-   **メモには必ず `path/to/file.ext:行` の参照** と 1 行の理由を添える。コードを貼るならフェンスで囲み、前後に説明を付ける。
-   **`### 残課題` は常に置く**（無ければ「なし」と明記）。**`### 意思決定の理由` は却下案も書く**。
-   **lint / test コマンドはプロジェクトの規約に従う**（以下の例では `npm run lint` / `go test ./...` 等を仮で置いている。実際は `CLAUDE.md` / `README` 記載のコマンドに置き換える）。

> 以下の各例は、見やすさのため 4 連バッククォート ` ```` ` でタスクファイル全体を囲んでいる（中のコードは通常の 3 連バッククォート）。実ファイルにはこの外側の囲みは無く、中身だけが `task.*.md` に入る。

````markdown
# fix: 「未入力」フィルターで一部の項目が一覧から漏れる

## 関連リンク

-   Issue: https://github.com/<owner>/<repo>/issues/123
-   PR: 未作成
-   関連 issue・サブタスク: なし（調査次第で派生 issue を立てるかも）

## チェックリスト

-   [x] 再現条件の特定（どの操作後に漏れるか）
-   [x] 「未入力」フィルターの絞り込みクエリ生成箇所を調査
-   [x] 原因の特定
-   [ ] 修正方針の確認（フィルター側で空判定を拡張する）
-   [ ] 絞り込みクエリの `id IS NULL` 条件を修正
-   [ ] 数値・文字列フィルターなど同種パターンに同じ穴がないか調査
-   [ ] テスト追加
-   [ ] lint / test 通過

---

## メモ

### 原因

「未入力」フィルターはフロント (`src/components/ListFilter.tsx:157-173`) で未入力項目を
絞り込むクエリを組み立てる。サーバ側の `applyDateSearch`
(`server/repository/search_repository.go:159-178`) が次の SQL を生成する:

```sql
LEFT JOIN field_values fv
  ON fv.record_id = records.id
 AND fv.field_name = 'target_date'
WHERE fv.id IS NULL
```

→ 「`field_values` に該当行が **存在しない**」レコードのみを空扱いしている。
あるステータス遷移で空の `target_date` 行（値 NULL）が作られると、行は存在するが値が NULL
の状態になり、`id IS NULL` を満たさず一覧から漏れる。これが root cause。

### 修正

修正案: 空判定を「行が無い」だけでなく「値が NULL」も含めるよう拡張する。

```go
// server/repository/search_repository.go:170 付近
return q.Joins(joinSQL, name).Where(
    fmt.Sprintf("fv_%s.id IS NULL OR fv_%s.value IS NULL", name, name),
)
```

（調査中）数値・文字列フィルターにも同じ `id IS NULL` 限定の穴がないか確認する。

### 検証

（未着手）

### 意思決定の理由

書き込み側で「空の行を作らない」案も検討したが、既存データに既に空行が存在しており
遡及修正が必要になるため却下。影響範囲を最小化できるフィルター側（読み取り側）の
修正に絞る。

### 残課題

-   数値・文字列フィルターの同種パターンは調査中。別 issue 化の可能性あり。
````
